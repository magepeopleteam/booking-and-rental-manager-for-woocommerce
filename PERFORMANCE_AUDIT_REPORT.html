3. Detailed Findings

1
Critical N+1: Monthly Calendar Loop — 31× Repeated Meta Fetches
Critical
File
inc/rbfw_inventory_functions.php
Lines
552 – 694 (function rbfw_day_wise_sold_out_check_by_month())
Type
N+1 Query — get_post_meta() inside day-iteration loop
Impact
31 iterations × 4 meta keys = 124 redundant DB queries per calendar render
Inside the day-of-month loop (lines 578–668), the function calls get_post_meta() four times per iteration for rbfw_enable_variations, rbfw_variations_stock, rbfw_item_type, and rbfw_inventory. These values are identical for all 31 days — the post ID never changes within the loop. With a 31-day month, this produces 124 duplicate database round-trips for metadata that could be fetched once.
Fix — Move 4 get_post_meta() calls above the loop (before line 578): Store results in local variables ($enable_variations, $variations_stock, $item_type, $inventory) and reference them inside the loop. Zero logic change, zero UI change. Expected saving: ~120 DB queries per calendar widget render.

Critical N+1: Inventory Page Table — 8+ Meta Calls Per Item
Critical
File
inc/rbfw_inventory_functions.php
Lines
773 – 1126 (function rbfw_inventory_page_table())
Type
N+1 Query — WP_Query(-1) + per-item meta calls in loop
Impact
500 items × 8 meta keys = 4,000+ DB queries on admin inventory page
Line 779 fetches ALL rental posts with posts_per_page => -1. Inside the foreach loop (lines 904–1080), 8 individual get_post_meta() calls are made per item for: rbfw_item_type, rbfw_enable_variations, rbfw_variations_data, rbfw_resort_room_data, rbfw_bike_car_sd_data, rbfw_extra_service_data, rbfw_inventory, and rbfw_service_category_price. WordPress internally caches post meta when update_postmeta_cache is true, but each call still traverses the internal cache lookup chain.
Fix — Two-part approach: (1) Replace posts_per_page => -1 with pagination (50 per page) to cap worst-case load. (2) Add 'update_post_meta_cache' => true explicitly to the WP_Query args (it is true by default but making it explicit future-proofs the query). This causes WordPress to batch-load ALL post meta for all retrieved posts in a single query, eliminating N+1.

High: Related Products — 300+ DB Queries per Page with 20 Items
High
File
inc/rbfw_functions.php
Lines
495 – 707 (function rbfw_related_products())
Type
get_post_meta() inside foreach loop — 15+ calls per related item
Impact
20 related items × 15 meta keys = 300 DB queries per product page
For every related product in the foreach loop, individual get_post_meta() calls are made for: rbfw_item_type, rbfw_gallery_images, rbfw_enable_hourly_rate, rbfw_hourly_rate, and all 7 day-rate keys (rbfw_mon_hourly_rate through rbfw_sun_hourly_rate), plus day-enable flags (rbfw_enable_sun_day through rbfw_enable_sat_day). This is exacerbated because the WP_Query fetching the related posts likely does NOT have update_post_meta_cache => true, so WordPress does not batch the meta.
Fix — Ensure the WP_Query for related posts includes 'update_post_meta_cache' => true. WordPress will then load all meta for all related items in one SQL query. Inside the loop, get_post_meta() will be served from the internal object cache. Zero logic change. Add a transient cache around the entire function output keyed by rbfw_related_{$post_id} with 1-hour expiry for further savings.

High: wp_cache_flush() on Every Admin Page Load
High
File
functions.php (or inc/rbfw_functions.php — rbfw_page_create())
Lines
~210–215
Type
wp_cache_flush() on admin_init hook — destroys entire object cache
Impact
Every admin page load invalidates all cached data (posts, terms, options, transients in memcached/Redis)
The rbfw_page_create() function, hooked to admin_init, calls wp_cache_flush() unconditionally. This destroys the entire WordPress object cache on every single admin page load, making object caching (Redis, Memcached) completely ineffective and causing significant performance degradation in environments that use a persistent cache.
Fix — Gate the flush behind a condition and replace with targeted delete: Only call wp_cache_delete() for the specific cache key (e.g., wp_cache_delete('rbfw_created_pages', 'rbfw')) and only when a page is actually being created. Remove the unconditional wp_cache_flush() entirely.

High: Inventory Functions — Duplicate Meta Within Same Function
High
File
inc/rbfw_inventory_functions.php
Lines
237, 240, 282 — function rbfw_get_multiple_date_available_qty()
Type
get_post_meta() for same key at top of function AND again inside loop
Impact

get_post_meta($post_id, 'rbfw_inventory', true) is called at line 240, result stored in $inventory. High
File
inc/rbfw_order_meta.php
Lines
606, 615, 667, 678, 726–783
Type
Sequential get_post_meta() calls — no batch pre-load
Impact
5+ individual DB calls per order meta box render in WP admin
In rbfw_order_meta_box_callback(), each meta key is fetched separately: rbfw_order_status (line 606), rbfw_ticket_total_price (615), rbfw_pickup_date (667), rbfw_return_date (678), and rbfw_ticket_info (726). A second foreach loop (lines 729–783) also calls get_post_meta() inside without caching.

High: Timely Availability — get_option() Inside Inventory Loop
High
File
inc/rbfw_functions.php
Lines
1238 – 1309 (function rbfw_timely_available_quantity_updated())
Type
get_option() (via rbfw_get_option()) called inside inventory loop
Impact
N repetitions of options DB lookup per availability check
Lines 1254–1262 call rbfw_get_option() (which internally calls get_option()) inside the inventory-iteration loop. While WordPress autoloads and caches options, calling inside a loop still adds function call overhead and prevents static analysis. More importantly, line 1161 calls get_option('mepp_reduce_stock', 'full') inside an inner loop in a related function.
Fix — Hoist all rbfw_get_option() / get_option() calls above their respective loops into local variables. Reference the variables inside. Applies to both rbfw_timely_available_quantity_updated() and rbfw_get_bike_car_sd_available_qty().

Medium: Fee Functions — Triple Identical Meta Fetch
Medium
File
inc/rbfw_fee_functions.php
Lines
29–40, 90–106, 120–152
Type
3 functions call rbfw_get_fee_data($item_id) independently
Impact
3 identical get_post_meta() calls when all 3 functions are invoked together
rbfw_get_active_fees(), rbfw_get_refundable_fees(), and rbfw_get_taxable_fees() each call rbfw_get_fee_data($item_id) which internally calls get_post_meta(). When cart price calculation invokes all three consecutively for the same item, the same meta key is fetched 3 times.
Fix — Add a static runtime cache inside rbfw_get_fee_data(): static $cache = []; if (isset($cache[$item_id])) return $cache[$item_id]; $cache[$item_id] = get_post_meta(...); return $cache[$item_id]; One function modified, all three callers instantly benefit. Zero behavior change.

Medium: Style Settings — get_option() Called 13× Per Page
Medium
File
inc/RBFW_Style.php
Lines
16 – 183 (function rbfw_style())
Type
RBFW_Function::get_style_settings() called 13 times per hook execution
Impact
Fires on both wp_head AND admin_head — 26 option reads per full page
Lines 17–33 call RBFW_Function::get_style_settings() 13 times consecutively for individual style keys (rbfw_theme_color, rbfw_font_size, etc.). Each call internally calls get_option('rbfw_basic_style_settings') and parses the returned array. WordPress caches autoloaded options in memory after the first load, so subsequent calls won't hit the DB — but the PHP-level array lookup and function call overhead still occurs 13×.
Fix — Call get_option('rbfw_basic_style_settings', []) once at the top of rbfw_style(), store in $settings, and access all 13 keys directly from that array. Reduces 13 PHP function call chains to 1. No output change.

Medium
File
rent-manager.php
Lines
58 – 62 (function add_body_class())
Type
Same get_post_meta() called twice on same line in ternary
Impact
Every product page: 1 redundant meta fetch in filter on body_class
Line 60 evaluates get_post_meta($post_id, 'rbfw_single_template', true) in the condition AND again in the true-branch of a ternary. PHP evaluates both sides when the condition is truthy, causing two identical meta calls.

Medium: Cart Duplicate Check — No Early Exit on Match
Medium
File
Frontend/RBFW_Woocommerse.php
Lines
23 – 94 (function rbfw_prevent_duplicate_cart_item())
Type
foreach loop with no break on found condition
Impact
Large carts (50+ items) iterate fully even after match is found
The function iterates through WC()->cart->get_cart() to check for duplicate rental dates. After finding a match and setting $found = true, the loop continues iterating through all remaining cart items instead of stopping. For carts with many items, this wastes CPU cycles proportional to cart size.
Fix — Add break; immediately after setting $found = true. One-line change, zero behavior change — the result is identical, just computed faster.
